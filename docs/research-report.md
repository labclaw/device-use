# Device-Use: AI Agent for Scientific Instrument Software
## Deep Research Report — 2026-03-09

---

## Executive Summary

**Core Thesis**: 构建一个专注科学研究领域的 computer-use 产品 —— "device-use"，让 AI agent 能够通过视觉理解和 GUI 操作来控制任意科学仪器软件，从而实现 AI 自主做实验。

**关键发现**: 这是一个真实存在的市场空白。当前生态中：
- Cloud Labs（ECL、Strateos）绕过现有仪器，要求样品寄送到集中设施
- 机器人自动化（Opentrons、Hamilton）控制液体处理，但不操作仪器分析软件的 GUI
- 数据平台（TetraScience、Ganymede）整合数据，但不操控仪器
- 通用 computer-use（GPT-5.4 75%、Claude 72.5%、Agent-S3 72.6%）技术已超越人类水平，但无人专注科学领域
- 自驱实验室（Kebotix、学术项目）是定制的、昂贵的、与硬件绑定的

**Device-use 填补的空白**: 一个能操作现有仪器软件 GUI 的 AI agent — 不需要替换硬件，不需要 API 集成，像训练有素的技术员一样看屏幕、操作仪器。

## 2. 技术架构深度分析

### 2.1 三种感知范式

```
┌─────────────────────────────────────────────────────────┐
│                   感知层对比                              │
├──────────────┬──────────────┬───────────────────────────┤
│   纯视觉      │  可访问性树    │       混合方案             │
├──────────────┼──────────────┼───────────────────────────┤
│ 截图 → VLM    │ OS API → 结构 │ 可访问性优先 → 视觉兜底      │
│ 任意应用可用   │ 依赖应用支持   │ 最佳覆盖率 + 可靠性         │
│ 高延迟/高成本  │ 快速/廉价     │ 中等成本                   │
│ 准确率: ~72%  │ 覆盖: ~33%   │ 最佳实践                   │
│              │ (macOS)      │                           │
└──────────────┴──────────────┴───────────────────────────┘
```

**对科学仪器的影响**: 大多数科学仪器软件是 Windows 原生应用，使用自定义 UI 框架（如 Qt、WinForms、WPF、甚至 MFC）。可访问性树覆盖率不确定，因此**纯视觉或混合方案是必须的**。

### 2.2 UI 元素定位（Grounding）技术

| 方法 | 原理 | 准确率 | 适用场景 |
|------|------|--------|----------|
| **坐标预测** (text_2_point) | VLM 输出 (x,y) 坐标 | 中等 | Anthropic 采用 |
| **边界框预测** (text_2_bbox) | VLM 输出四角坐标 | 中等偏上 | 更宽容的定位 |
| **Set-of-Mark (SoM)** | 叠加标记 → VLM 引用标签 | 高（zero-shot > finetuned SOTA） | 无原生 grounding 能力的模型 |
| **注意力机制** (GUI-Actor) | 图像 patch 正/负标注 | **最高**（7B 超过 72B） | 前沿研究 |
| **OmniParser 管线** | 检测→描述→OCR | GPT-4o: 0.8%→39.6% | 任意 LLM 的增强 |

**ScreenSpot-Pro 基准**（专业高分辨率桌面软件）:
- 原始 GPT-4o: 0.8%
- OmniParser + GPT-4o: 39.6%
- Qwen3-VL-32B: 74.0%
- SE-GUI-7B: 88.2%（ScreenSpot）

### 2.3 动作执行技术栈

| 层级 | 工具 | 平台 | 说明 |
|------|------|------|------|
| **通用输入模拟** | PyAutoGUI | 跨平台 | 事实标准，有 MCP server |
| **平台原生 API** | Win32/UI Automation | Windows | 更可靠，但需要可访问性支持 |
| **平台原生 API** | AppleScript | macOS | 可脚本化应用 |
| **平台原生 API** | D-Bus/AT-SPI | Linux | 较少标准化 |
| **仪器通信** | PyVISA/SCPI | 跨平台 | 测量仪器直接通信 |

### 2.4 关键技术挑战

| 挑战 | 严重度 | 现有解决方案 | 对 device-use 的影响 |
|------|--------|-------------|---------------------|
| **延迟** | 高 | OmniParser V2 减少 60%；每步仍 3-10s | 科学仪器通常不需要实时响应，可接受 |
| **多步骤错误复合** | 极高 | 90% 单步准确率 × 20 步 = 12% 成功率 | 需要 checkpoint/rollback 机制 |
| **错误恢复** | 高 | UI-TARS 反思调优；AutoDW 回滚 | 科学实验中错误可能浪费昂贵试剂 |
| **自定义 UI 框架** | 中高 | 纯视觉是唯一选项 | 仪器软件大量使用自定义控件 |
| **高分辨率小元素** | 中高 | 目标元素仅占屏幕 0.07% | 仪器软件界面复杂密集 |
| **状态验证** | 中 | 静态评估无法捕捉动态变化 | 需要理解仪器状态（采集中/完成/错误）|

## 3. 科学仪器自动化现状

### 3.1 科学仪器软件生态

| 仪器类型 | 控制软件 | 厂商 | 自动化能力 |
|----------|---------|------|-----------|
| 共聚焦/光学显微镜 | ZEN (Blue/Black) | ZEISS | IronPython 脚本、TCP/IP 远程控制 |
| 显微镜 | NIS-Elements | Nikon | JavaScript 脚本、图形化工作流 |
| FTIR/拉曼光谱仪 | OPUS | Bruker | 宏语言，有限 API |
| 流式细胞仪 | FlowJo, FACSDiva | BD/Cytek | 批处理、模板，有限脚本 |
| 液相色谱 | Empower, OpenLab | Waters/Agilent | 序列自动化，部分脚本 |
| 酶标仪 | SoftMax Pro, Gen5 | Molecular Devices | 协议模板，数据导出 |
| qPCR | QuantStudio, CFX Maestro | ABI/Bio-Rad | 实验模板，有限 API |
| 质谱 | MassLynx, Xcalibur | Waters/Thermo | 序列控制，脚本接口 |

### 3.2 核心痛点（Why device-use matters）

```
科学家的日常：

1. 打开仪器软件 A → 手动设置参数 → 点击"开始采集"
2. 等待完成 → 手动导出数据（专有格式）
3. 打开分析软件 B → 手动导入数据 → 手动处理
4. 复制结果到 Excel/ELN
5. 重复以上步骤 50-200 次

痛点根源：
├── 20+ 不同软件包，各有自己的界面和数据格式
├── 厂商锁定：专有二进制格式，跨厂商数据无法流通
├── 自动化需要编程 → 大多数实验科学家不会编程
├── 遗留系统：很多仪器运行在 Windows 7/XP 上
├── 每台仪器集成费用 $5K-$65K
└── 无跨仪器编排：多步实验需要人工在每台仪器间传递
```

### 3.3 现有自动化标准和协议

| 标准/工具 | 说明 | 采用率 | 局限 |
|-----------|------|--------|------|
| **SiLA 2** | gRPC 实验室通信协议 | 低，增长缓慢 | 多数仪器不实现 |
| **OPC UA LADS** | 工业 OPC UA 的实验室扩展 | 极低（新标准） | 仅新仪器可能支持 |
| **PyVISA/SCPI** | 测量仪器通信 | 中（测量类） | 仅限 GPIB/USB/串口仪器 |
| **厂商 API** | ZEN OAD, NIS-Elements JS | 中（各自生态） | 各不相同，互不兼容 |
| **开源框架** | Bluesky, PLACE, PyOpticon | 低（学术） | 需要编程能力 |

**关键洞察**: 标准采用太慢，API 各自为政 — GUI 是唯一真正通用的接口。每个仪器，无论多旧多封闭，都有 GUI。

### 3.4 自驱实验室（Self-Driving Labs）现状

| 项目/公司 | 模式 | 融资 | 局限 |
|-----------|------|------|------|
| **Emerald Cloud Lab** | 集中云实验室 | 大规模 | >$250K/年，需寄送样品 |
| **Strateos** | 云实验室（已转型） | 被收购 | 公共模式难以商业化 |
| **Lila Sciences** | AI 科学工厂 | $550M | 全定制硬件+软件 |
| **Periodic Labs** | AI 科学家 | $300M seed | 建设自己的机器人高通量实验室 |
| **Edison Scientific** | FutureHouse 商业化 | $70M seed | 聚焦假说生成，不控制仪器 |

## 5. 监管考量

### 5.1 关键法规

| 法规 | 适用场景 | 对 device-use 的要求 |
|------|----------|---------------------|
| **FDA 21 CFR Part 11** | 制药/生物科技 | 不可变审计轨迹、电子签名、角色访问控制、系统验证 |
| **GLP/GMP** | 受管制实验室 | 数据完整性（ALCOA+）、不可关闭的审计轨迹 |
| **EU Annex 11** | 欧洲制药 | 计算机系统验证、数据完整性 |

### 5.2 合规策略

这是**护城河也是壁垒**：
- **壁垒**: 系统验证（IQ/OQ/PQ）成本高，需要文档化 AI 算法版本、人工审核机制
- **护城河**: 一旦通过验证，替换成本极高；竞争者需重新走完验证流程
- **建议**: 第一阶段面向"仅供研究使用"(RUO) 场景（学术/早期研发），避免合规负担；第二阶段推出 GxP-ready 版本

## 6. 商业模式建议

### 6.1 定价模型对比

| 模型 | 优点 | 缺点 | 适合阶段 |
|------|------|------|----------|
| **按操作次数** (per-action) | 与价值对齐；低门槛 | 难预测收入 | 早期增长 |
| **按仪器/设备** (per-device) | 直觉清晰；可预测 | 限制扩展 | 中期 |
| **按座位** (per-seat) | 行业惯例 | AI 时代逐渐过时 | 可选 |
| **平台订阅 + 用量** | 可预测基础 + 增长空间 | 定价复杂 | 成熟期 |

**推荐**: 按仪器/设备订阅 + 超额用量计费。每台仪器 $200-500/月（CRO/企业），学术免费或 $50/月。

### 6.2 开源 vs 闭源策略

**推荐混合模式**:
- **开源核心**: 基础 GUI 操作引擎、通用仪器交互框架 → 社区贡献仪器配置
- **闭源商业层**: 合规审计轨迹、企业管理、高级工作流编排、SLA 支持
- **类似模式**: Opentrons（开源软件 + 硬件销售）、GitLab（开源核心 + 企业版）

### 7.3 核心差异化: Science-Aware Reasoning

通用 computer-use 和 device-use 的核心区别在于**科学语义理解**:

| 通用 Computer Use | Device-Use (Science-Aware) |
|-------------------|---------------------------|
| "点击这个按钮" | "设置激光功率为 10%，因为这种荧光染料容易光漂白" |
| "在文本框输入 42" | "将曝光时间调整为 42ms，并验证直方图未饱和" |
| "截图保存" | "导出 TIFF 格式的 z-stack 数据，保留元数据" |
| 不理解操作上下文 | 理解这是在做免疫荧光成像，知道操作顺序和约束 |

构建这种语义理解的方法:
1. **仪器知识库**: 每种仪器软件的操作手册、工作流程、参数含义
2. **领域 SOP 库**: 标准操作规程的结构化表示
3. **科学文献 RAG**: 检索增强生成，关联实验方案
4. **用户反馈循环**: 从科学家的纠正中持续学习

## 8. 风险与挑战

| 风险 | 等级 | 缓解策略 |
|------|------|----------|
| **准确率不足导致实验浪费** | 高 | 关键步骤 human-in-the-loop；操作前预览确认 |

| **仪器损坏风险** | 高 | 安全边界检查；禁止超出参数范围的操作 |
| **数据安全/隐私** | 高 | 本地部署选项；不发送实验数据到外部 |
| **监管合规负担** | 中 | 初期 RUO only；逐步建设 GxP 能力 |
| **竞争者进入** | 中 | 先发优势 + 仪器知识库 + 社区壁垒 |

## 12. CUA-Bench 深度分析 — 基础设施启示

### 12.1 CUA-Bench 是什么

[CUA-Bench](https://cuabench.ai/) 由 [Cua AI](https://cua.ai/)（YC 孵化）开发，是一个 computer-use agent 的基准测试和训练数据生成框架。它解决了一个核心问题：**当前 agent 在轻微 UI 变化下性能波动高达 10 倍**。

| 维度 | CUA-Bench | OSWorld | WindowsAgentArena |
|------|-----------|---------|-------------------|
| **环境启动** | 轻量 Webtop（秒级） | VM 快照（~20分钟） | VM（慢） |
| **应用灵活性** | 可配置、可随机化 | 固定预装 | 固定预装 |
| **任务定义** | Python Decorator API（灵活） | JSON（受限） | 结构化 |
| **视觉多样性** | 自动生成（主题/分辨率/时代） | 无 | 无 |
| **Ground-Truth 轨迹** | 完整捕获（HTML+截图+输入） | 无 | 无 |
| **场景变体** | JSON 注入→单模板生成千变体 | 手动 | 手动 |
| **跨平台** | macOS/Linux/Windows/iOS/Android/VM/Webtop | Ubuntu/Windows | Windows |

### 12.2 CUA-Bench 排行榜（cua-bench 2.0）

| Agent/Model | 成功率 |
|-------------|--------|
| **Claude Haiku 4.5** | **68.4%** |
| Claude Sonnet 4.5 | 59.1% |
| UI-TARS-2 | 58.0% |
| OpenAI CUA (GPT-5.x) | 57.8% |
| Gemini CUA | 54.2% |

**洞察**: Haiku 4.5 领先 Sonnet 4.5 近 10 个百分点 — 可能因为 CUA-Bench 侧重 UI 多样性和鲁棒性，而非复杂推理。这对 device-use 的启示是：**在科学仪器场景，鲁棒性（不因 UI 变化失败）可能比原始推理能力更重要**。

### 12.3 CUA-Bench 的三大核心创新

#### 1. 可扩展 GUI 数据生成
```
单个任务模板 → JSON 场景注入 → 数千变体
                               ├── 不同主题（暗色/亮色/高对比度）
                               ├── 不同分辨率（640x480 → 3440x1440）
                               ├── 不同窗口布局
                               ├── 不同 OS 版本（Win98 → Win11）
                               └── 不同语言设置
```

**对 device-use 的启示**: 科学仪器软件有大量 UI 变体 — ZEN Blue vs Black, OPUS 不同版本, FlowJo 10 vs CE。我们需要类似的变体生成能力来确保跨版本鲁棒性。

#### 2. 轨迹重绘（Trajectory Replotting）
```
录制 1 次人工演示 → 在 10 个 OS 主题下重新渲染 → 10 条训练轨迹
相同动作序列，不同视觉表现 = 跨平台鲁棒性
```

**对 device-use 的启示**: 这是构建仪器软件训练数据的关键技术。一个科学家在 ZEN Blue 上录制一次操作，系统自动生成适配 ZEN Black、不同版本、不同屏幕分辨率的训练数据。

#### 3. Shell Applications（模拟应用）
CUA-Bench 提供了 Spotify/Slack/WhatsApp 的全功能克隆，agent 可以在无需真实 VM 的环境中训练和测试。

**对 device-use 的关键启示**: 我们可以构建 **仪器软件模拟器**（Instrument Shell Apps）:
- ZEN 模拟器：模拟显微镜控制界面
- OPUS 模拟器：模拟光谱仪操作
- FlowJo 模拟器：模拟流式分析

这解决了训练数据获取的核心难题 — 不需要访问真实仪器就能生成大量训练数据。

### 12.4 Cua 平台 vs Device-Use 定位

| 维度 | Cua (cua.ai) | Device-Use |
|------|-------------|------------|
| **定位** | "Docker for Computer-Use Agents"（通用基础设施） | "AI Brain for Scientific Instruments"（垂直应用） |
| **范围** | 任意桌面应用 | 科学仪器软件 |
| **差异化** | 沙箱/容器化/多平台 | 科学语义理解/仪器知识库/实验闭环 |
| **客户** | AI 开发者/研究者 | 科学家/实验室 |
| **商业模式** | 云计算信用点/按时计费 | 按仪器订阅 |
| **开源** | MIT 许可（13K stars） | 开源核心（计划） |
| **关系** | **潜在技术合作伙伴** | 在 Cua 基础设施上构建垂直应用 |

**关键洞察**: Cua 是通用基础设施层，Device-Use 是垂直应用层。最佳策略不是与 Cua 竞争，而是**在 Cua 之上构建** — 用 Cua 的沙箱和 SDK 作为底层，在上层构建科学仪器的领域知识、SOP 库和实验闭环。

### 16.4 Medra 深度剖析 — 最接近的竞品

**Medra vs Device-Use 核心差异**:

| 维度 | Medra | Device-Use |

| **成本** | $52M 融资→重资产模式 | 轻资产，按月订阅 |

| **边界** | 物理操作（移液、放样品） + 软件操作 | 软件操作为主 |
| **适用场景** | 需要物理操作的湿实验 | 仪器控制/数据采集/分析 |
| **开源** | 闭源 | 开源核心 |
| **关系** | **互补而非竞争** | 可以和 Medra 的机器人协同 |

**关键洞察**: Medra 的机器人解决的是"拿起移液器加样品"的物理操作，Device-Use 解决的是"在仪器软件上点击开始采集"的软件操作。两者解决实验闭环的不同层面 — **理想的未来是 Medra 的机器人 + Device-Use 的软件控制协同工作**。

### 16.5 Scaling Laws 论文的启示

arXiv 2503.22444 提出了 **Autonomous Generalist Scientist (AGS)** 的概念和科学发现的缩放定律:

- **公式**: P = αR^k （P=发现数，R=资源，k=0.1-0.8）
- **核心预测**: Robot Scientists 可以每天设计和执行 10,000+ 实验
- **过渡期**: 2025-2040，AGS 系统逐步补充并最终超越人类研究产出
- **关键组件**: agentic AI + embodied robotics，覆盖文献→假说→实验→论文

**对 Device-Use 的启示**: 这个缩放定律指出，当实验执行成本趋近零时，科学发现会爆发式增长。Device-Use 的价值正是**将实验执行的边际成本从"一个训练有素的技术员的时间"降低到"API 调用成本"**。
